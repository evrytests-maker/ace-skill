## 1. Правила выбора ключевых слов

Ключевые слова — это триггеры, которые определяют, когда конкретная запись лорбука будет вставлена в контекст. Правильный выбор ключей напрямую влияет на точность и релевантность подачи лора.

### 1.1. Основные принципы

- **Ключи должны соответствовать содержанию и смыслу записи.** Если запись описывает "цветочные поля", ключи должны быть связаны именно с этой темой — не с окружающими концепциями, не с персонажами, которые там находятся.
- **Учитывайте язык ролеплея.** Для русскоязычного ролеплея ключи должны быть на русском языке. Для английского — на английском. Смешение языков в ключах одной записи без необходимости создаёт лишний шум.
- **Отдавайте предпочтение ОДНОМУ слову вместо фраз.** Фразы (состоящие из нескольких слов) имеют МЕНЬШУЮ вероятность срабатывания из-за зависимости от порядка слов в чате. Если бот напишет "поля, покрытые цветами" — ключ "цветочные поля" НЕ сработает, потому что слова стоят в другом порядке и с другими вставками.
- **Количество ключей — умеренное.** Оптимально ~10 ключевых слов на запись. Иногда достаточно 5. Большое количество ключей (15-20+) перегружает систему и повышает вероятность ложных срабатываний.
- **Планируйте ключи при рекурсии с конца цепочки к началу.** Если запись A вызывает запись B, которая вызывает запись C — сначала определите ключи для C, затем для B (которые сработают при активации C), затем для A. Это обратное планирование предотвращает зацикливание и неожиданные каскадные активации.

### 1.2. Что НЕЛЬЗЯ использовать в качестве ключей

| Категория | Пример | Почему нельзя |
|---|---|---|
| Местоимения | он, она, его, её | Слишком частотные — срабатывают в каждом сообщении, полностью обесценивают триггер |
| Предлоги | в, на, под, за | Появляются буквально в каждом предложении, создают массовые ложные срабатывания |
| Короткие корни, входящие в состав других слов | "лес" (лесной, лесок, лесничий) | Срабатывают на производные формы даже при Whole Words = ON из-за особенностей морфологии |
| Имена пользователя / бота | никнейм | Срабатывают при каждом обращении, не несут смысловой нагрузки для лора |
| Символы из HTML/CSS/блоков мыслей | <div>, <thinking>, {{char}} | Технические блоки тоже сканируются на ключи — триггер может сработать на скрытом системном тексте |

> **Критически важно:** HTML-разметка, CSS-стили и блоки мышления (thought blocks) также участвуют в сканировании ключей. Ключ, совпадающий с тегом или системным маркером, будет срабатывать скрытно, без видимой причины.

### 1.3. Почему порядок слов важен для фраз

SillyTavern сканирует ключи как подстроки (или целые слова, в зависимости от настройки). Ключ-фраза "цветочные поля" ищет именно эту последовательность символов подряд. Если в чате написано:

- "поля цветочные" — НЕ сработает (другой порядок)
- "цветочные луга и поля" — НЕ сработает как целая фраза (есть разрыв)
- "по цветочным полям" — зависит от подхода (суффиксный захват или regex)

Поэтому для фразовых ключей критически рекомендуется использовать regex-подход с гибким паттерном.

## 2. Три подхода к составлению ключей

Для одной и той же записи существуют три технически различных стратегии составления ключей. Каждая имеет свои настройки, преимущества и ограничения.

### 2.1. Общая тестовая запись для примеров

Для демонстрации всех трёх подходов используем запись, описывающую локацию:

**Content:** `Цветочные поля за деревней, усыпанные незабудками и маками. Кот Никс часто греется здесь при закатных лучах.`

Целевые ключевые концепции: цветочные поля, незабудки, маки, закатные лучи, кот Никс.

---

### 2.2. Подход 1: Простой (все грамматические формы)

**Принцип:** Перечислить ВСЕ возможные грамматические формы каждого слова вручную. Система ищет точное совпадение целого слова.

**Настройки активации:**

| Параметр | Значение | Обоснование |
|---|---|---|
| Case Sensitive | OFF | Игнорировать регистр (цветочные / Цветочные / ЦВЕТОЧНЫЕ) |
| Whole Words | ON | Искать только целые слова, исключая вхождение внутри других слов |

**Ключи для тестовой записи (Подход 1):**

```
цветочные, цветочное, цветочного, цветочному, цветочным, цветочном,
цветочная, цветочной, цветочную,
цветочный, цветочного, цветочному, цветочным, цветочном,
цветочное, цветочного, цветочному, цветочным, цветочном,
цветочные, цветочных, цветочным, цветочными, цветочных,
поле, поля, полю, полем, поле,
полей, полям, полями, полях,
незабудка, незабудки, незабудке, незабудку, незабудкой, незабудке,
незабудки, незабудок, незабудкам, незабудками, незабудках,
мак, мака, маку, маком, маке,
маки, маков, макам, маками, маках,
закатный, закатного, закатному, закатным, закатном,
закатная, закатной, закатную,
закатные, закатных, закатным, закатными, закатных,
луч, луча, лучу, лучом, луче,
лучи, лучей, лучам, лучами, лучах,
кот, кота, коту, котом, коте,
коты, котов, котам, котами, котах,
никс, никса, никсу, никсом, никсе
```

**Преимущества:**
- Максимальная надёжность — нет ложных срабатываний
- Простота реализации — не требуется знание regex
- Предсказуемое поведение

**Недостатки:**
- Огромное количество ключевых слов (для примера выше — 70+ форм)
- Ручная работа по перечислению всех форм
- При добавлении новых слов в описание — необходимо расширять список форм
- Перегрузка интерфейса длинным списком ключей

**Оптимизация:** Комбинируйте с Secondary Filters + логикой AND ANY, чтобы сократить количество ключей за счёт условий срабатывания.

---

### 2.3. Подход 2: Отсечение окончаний (суффиксный захват)

**Принцип:** Указывать только корень слова без окончаний. Система ищет подстроку внутри любого слова.

**Настройки активации:**

| Параметр | Значение | Обоснование |
|---|---|---|
| Case Sensitive | OFF | Игнорировать регистр |
| Whole Words | OFF | Поиск подстрок — корень может встречаться внутри слова |

**Ключи для тестовой записи (Подход 2):**

```
цветоч, пол, незабуд, мак, закатн, луч, кот, никс
```

Всего 8 ключей вместо 70+.

**Преимущества:**
- Минимальное количество ключей
- Быстрое составление
- Один ключ покрывает все формы слова автоматически

**Недостатки:**
- **Ложные срабатывания.** "пол" входит в "политика", "поле", "полный", "аполлон". "мак" входит в "макароны", "маклер", "макет". "кот" входит в "скот", "котлета", "который".
- Нет контроля над границами слова
- При неаккуратном выборе корня можно захватить десятки нерелевантных слов

**Митигация:** Комбинируйте с Secondary Filters (логика NOT ANY или NOT ALL), чтобы исключить конкретные ложные срабатывания. Например, добавить в Secondary с логикой NOT ANY: `политика, полный, макароны, маклер, скот`.

---

### 2.4. Подход 3: Regex (оптимальный, наиболее точный)

**Принцип:** Использовать регулярные выражения JavaScript для точного поиска всех форм слова за один паттерн.

**Настройки активации:**

| Параметр | Значение | Обоснование |
|---|---|---|
| Case Sensitive | OFF (для EN) | Флаг `/i` в regex обрабатывает регистр |
| Whole Words | ON | Границы слова контролируются самим regex (lookaround) |

> Для русского языка границы слова `\b` НЕ работают корректно, поэтому используются lookahead/lookbehind с символьными классами.

**Ключи для тестовой записи (Подход 3):**

```
(?:^|[^а-яёА-ЯЁ])цветочн[а-яё]*(?=[^а-яёА-ЯЁ]|$)
(?:^|[^а-яёА-ЯЁ])незабуд[а-яё]*(?=[^а-яёА-ЯЁ]|$)
(?:^|[^а-яёА-ЯЁ])мак[а-яё]*(?=[^а-яёА-ЯЁ]|$)
(?:^|[^а-яёА-ЯЁ])закатн[а-яё]*(?=[^а-яёА-ЯЁ]|$)
(?:^|[^а-яёА-ЯЁ])луч[а-яё]*(?=[^а-яёА-ЯЁ]|$)
(?:^|[^а-яёА-ЯЁ])кот[а-яё]*(?=[^а-яёА-ЯЁ]|$)
(?:^|[^а-яёА-ЯЁ])никс[а-яё]*(?=[^а-яёА-ЯЁ]|$)
```

> Для "поле / поля" паттерн сложнее из-за чередования корня: `(?:^|[^а-яёА-ЯЁ])пол[ьяеюйа-яё]*(?=[^а-яёА-ЯЁ]|$)` — но требует внимания к ложным срабатываниям.

**Преимущества:**
- Минимальное количество паттернов (по одному на корень)
- Точное покрытие ВСЕХ форм без ложных срабатываний (при правильном lookaround)
- Контроль границ слова через символьные классы
- Возможность обрабатывать фразы с учётом всех падежных вариаций

**Недостатки:**
- Требуется знание синтаксиса JavaScript regex
- Ошибка в паттерне может привести к полной неработоспособности ключа
- Необходимо тестировать паттерны перед применением

## 3. Regex в ключевых словах

### 3.1. Общие правила

- Regex-паттерны вводятся в поле ключевых слов точно так же, как обычные слова — через запятую.
- Между паттернами ставится запятая, как между обычными ключами.
- SillyTavern использует **JavaScript regex engine** — все паттерны должны соответствовать синтаксису JS RegExp.
- Regex может искать целые предложения со знаками препинания (запятые, точки, вопросительные знаки).
- Паттерн может содержать пробелы — они трактуются как часть regex.

### 3.2. Ввод regex в интерфейсе

```
// Пример ввода в поле Keywords (через запятую):
(?:^|[^а-яёА-ЯЁ])цветочн[а-яё]*(?=[^а-яёА-ЯЁ]|$),
(?:^|[^а-яёА-ЯЁ])незабуд[а-яё]*(?=[^а-яёА-ЯЁ]|$),
(?:^|[^а-яёА-ЯЁ])мак[а-яё]*(?=[^а-яёА-ЯЁ]|$),
закат
```

### 3.3. JS Regex для фраз со знаками препинания

Если нужно искать фразу, содержащую запятую или точку — экранируйте спецсимволы:

```javascript
// Поиск фразы "цветочные поля, усыпанные"
/(?:^|[^а-яёА-ЯЁ])цветочные поля, усыпанные(?=[^а-яёА-ЯЁ]|$)/i

// Поиск фразы с вопросительным знаком
/(?:^|[^а-яёА-ЯЁ])ты видел поля\?(?=[^а-яёА-ЯЁ]|$)/i
```

> Запятая в regex ВНУТРИ паттерна НЕ является разделителем ключей — SillyTavern различает запятую внутри regex (как часть паттерна) и запятую как разделитель по контексту. Тем не менее, сложные паттерны с запятыми рекомендуется тщательно тестировать.

## 4. Примеры ключей для русского языка

### 4.1. Разбор regex-паттерна для русского слова

Стандартный шаблон для покрытия всех форм русского слова:

```
(?:^|[^а-яёА-ЯЁ])КОРЕНЬ[а-яёА-ЯЁ]*(?=[^а-яёА-ЯЁ]|$)
```

| Компонент | Значение |
|---|---|
| `(?:^|[^а-яёА-ЯЁ])` | Non-capturing group: либо начало строки, либо символ НЕ из кириллицы. Это lookbehind-заменитель — проверяет, что перед словом нет буквы. |
| `КОРЕНЬ` | Статическая часть слова (корень + суффикс, которые не меняются). Например: "цветочн", "незабуд", "мак". |
| `[а-яёА-ЯЁ]*` | Ноль или более кириллических символов — покрывает любые окончания и суффиксы (ые, ое, ая, ий, ов, ам...). |
| `(?=[^а-яёА-ЯЁ]|$)` | Positive lookahead: после слова должен идти либо НЕ-буква, либо конец строки. Это гарантирует, что мы захватили целое слово, а не его часть. |

### 4.2. Полный набор паттернов для тестовой записи

| Концепция | Regex-паттерн |
|---|---|
| Цветочные поля | `(?:^|[^а-яёА-ЯЁ])цветочн[а-яё]*(?=[^а-яёА-ЯЁ]|$)` |
| Поле / поля | `(?:^|[^а-яёА-ЯЁ])пол[ьяеюйа-яё]*(?=[^а-яёА-ЯЁ]|$)` — требует осторожности |
| Незабудки | `(?:^|[^а-яёА-ЯЁ])незабуд[а-яё]*(?=[^а-яёА-ЯЁ]|$)` |
| Маки | `(?:^|[^а-яёА-ЯЁ])мак[а-яё]*(?=[^а-яёА-ЯЁ]|$)` |
| Закатные лучи | `(?:^|[^а-яёА-ЯЁ])закатн[а-яё]*(?=[^а-яёА-ЯЁ]|$)` и `(?:^|[^а-яёА-ЯЁ])луч[а-яё]*(?=[^а-яёА-ЯЁ]|$)` |
| Кот Никс | `(?:^|[^а-яёА-ЯЁ])кот[а-яё]*(?=[^а-яёА-ЯЁ]|$)` и `(?:^|[^а-яёА-ЯЁ])никс[а-яё]*(?=[^а-яёА-ЯЁ]|$)` |

### 4.4. Шаблон промпта для генерации regex через ИИ

Приведённый в PDF шаблон для автоматической генерации regex-паттернов:

```
Помоги составить JavaScript regex паттерны для ключевых слов lorebook.

Текст записи: [ВСТАВЬТЕ ТЕКСТ ЗАПИСИ]

Требования:
1. Каждый паттерн должен покрывать ВСЕ грамматические формы слова на русском языке (падежи, числа, род).
2. Использовать lookaround вместо \b (границы слова \b не работают с кириллицей).
3. Формат паттерна: (?<=^|[^а-яёА-ЯЁ])КОРЕНЬ[а-яё]*(?=[^а-яёА-ЯЁ]|$)
4. Перечислить паттерны через запятую.
5. Добавить флаг /i для игнорирования регистра.
6. Исключить местоимения, предлоги и частые слова.
7. Сфокусироваться на конкретных именах, терминах, топонимах и уникальных сущностях.

Ключевые концепции для покрытия: [ПЕРЕЧИСЛИТЕ КОНЦЕПЦИИ]
```

> Альтернативная форма с positive lookbehind `(?<=...)`: `(?<=^|[^а-яёА-ЯЁ])цветочн[а-яё]*(?=[^а-яёА-ЯЁ]|$)` — функционально эквивалентна `(?:^|[^а-яёА-ЯЁ])...`, но использует фиксированную ширину lookbehind. В JS lookbehind поддерживается в современных движках, но non-capturing group более совместим.

## 5. Вторичные фильтры (Secondary Filters)

Secondary Filters расположены справа от Primary Keywords в интерфейсе редактирования записи lorebook. Они позволяют уточнить условия срабатывания первичных ключей.

### 5.1. Принцип работы

Secondary Filters работают как **дополнительный слой проверки** поверх Primary Keywords. Запись активируется только если:
1. Хотя бы один Primary Keyword сработал (базовое условие), И
2. Условие Secondary Filter выполнено (дополнительное условие).

Secondary keywords используют те же свойства, что и Primary (Case Sensitive, Whole Words), но логика их применения определяется выбором оператора.

### 5.2. Четыре режима логики

#### 5.2.1. AND ANY (И хотя бы один)

**Условие:** Запись активируется, если сработал хотя бы один Primary Keyword **И** хотя бы один Secondary Filter.

**Когда использовать:** Для сужения области срабатывания. Primary задаёт общую тему, Secondary — конкретный подконтекст.

**Пример:**
- Primary: `цветочные, поля, маки, незабудки` (общая тема: цветочные поля)
- Secondary: `весна, апрель, май, весенн` (логика: AND ANY)
- **Результат:** Лор сработает только если в чате упоминаются цветочные поля И контекст весенний. Если кто-то скажет "цветочные поля зимой" — Primary сработает, но Secondary нет, и запись НЕ вставится.

#### 5.2.2. AND ALL (И все)

**Условие:** Запись активируется, если сработал хотя бы один Primary Keyword **И** ВСЕ Secondary Filters одновременно.

**Когда использовать:** Для очень специфических контекстов, где нужно наличие нескольких дополнительных маркеров.

**Пример:**
- Primary: `кот никс` (имя персонажа)
- Secondary: `закат, лучи` (логика: AND ALL)
- **Результат:** Лор сработает только если упомянут кот Никс И закат И лучи одновременно в зоне сканирования. Это крайне узкая активация.

#### 5.2.3. NOT ALL (НЕ все)

**Условие:** Запись активируется, если сработал Primary Keyword, но **НИ ОДИН** из Secondary Filters НЕ сработал.

**Когда использовать:** Для исключения конкретных контекстов. Активировать запись ТОЛЬКО если отсутствуют определённые слова.

**Пример:**
- Primary: `поле, поля` (общий ключ)
- Secondary: `битва, сражение, война` (логика: NOT ALL)
- **Результат:** Лор сработает при упоминании полей, но ТОЛЬКО если в контексте НЕТ боевых действий. "Цветочные поля" — активирует, "поле битвы" — не активирует.

#### 5.2.4. NOT ANY (НЕ любой)

**Условие:** Запись НЕ активируется, если сработал хотя бы один Primary Keyword **И** хотя бы один Secondary Filter. Иначе говоря: предотвращает активацию при наличии Secondary.

**Когда использовать:** Для блокировки активации в нежелательных контекстах. Мягче, чем NOT ALL — блокирует при наличии ЛЮБОГО secondary ключа.

**Пример:**
- Primary: `кот, кошка` (общий ключ)
- Secondary: `никс` (логика: NOT ANY)
- **Результат:** Общий лор о котах сработает для "кот", "кошка", но НЕ сработает если упомянут "никс". Это разделение: общий лор о котах vs. специфический лор о коте Никсе.

### 5.3. Сводная таблица логики

| Режим | Primary сработал | Secondary сработал | Результат | Применение |
|---|---|---|---|---|
| AND ANY | Да | Хотя бы один | Активация | Сужение контекста |
| AND ALL | Да | Все одновременно | Активация | Очень специфический случай |
| NOT ALL | Да | Ни один | Активация | Исключение нежелательных контекстов |
| NOT ANY | Да | Хотя бы один | БЛОКИРОВКА | Предотвращение пересечения с другим лором |

### 5.4. Настройки Secondary Filters

- **Case Sensitive:** работает независимо от Primary — можно задать свой регистр.
- **Whole Words:** работает независимо — можно использовать подстроки для Secondary даже при точном поиске Primary.
- Secondary Filters также поддерживают **regex-паттерны** — вводятся через запятую в поле Secondary Keywords.

### 5.5. Практический пример с Secondary Filters

```
Запись: "Оружие Тёмного Лорда"
Content: Тёмный Лорд владеет мечом Погибели, выкованным в адском пламени.

Primary Keywords: тёмный лорд, меч погибели, адское пламя
  Case Sensitive: OFF
  Whole Words: ON

Secondary Keywords: магия, заклинание, посох
  Logic: NOT ANY

Результат:
- "Тёмный Лорд замахнулся мечом" → АКТИВИРУЕТСЯ (Primary сработал, Secondary нет)
- "Тёмный Лорд произнёс заклинание" → НЕ активируется (Secondary "заклинание" блокирует)
- "Тёмный Лорд поднял посох" → НЕ активируется (Secondary "посох" блокирует)
```

## 6. Блок настроек активации (Activation Settings)

Эти параметры расположены в нижней части редактора записи lorebook и определяют механику сканирования и срабатывания.

### 6.1. Scan Depth (Глубина сканирования)

**Определение:** Количество последних сообщений в чате, которые сканируются на наличие ключевых слов.

| Значение | Значение |
|---|---|
| 1 | Сканировать ТОЛЬКО последнее сообщение |
| 2 | Сканировать 2 последних сообщения |
| 3 | Сканировать 3 последних сообщения |
| 4 (default) | Сканировать 4 последних сообщения |
| N | Сканировать N последних сообщений |
| 0 или All | Сканировать ВСЕ сообщения в чате |

**Рекомендации:**
- **Scan Depth = 1** для лора, который должен реагировать ТОЛЬКО на текущее сообщение (например, описание действия в ответ на конкретную реплику).
- **Scan Depth = 2-4** для стандартных записей — позволяет учитывать контекст предыдущих сообщений.
- **Scan Depth > 4** для важного фонового лора, который должен оставаться активным в течение долгой сцены.
- **Scan Depth = 0 (All)** для глобального лора, который должен проверяться по всей истории чата (например, мировые константы, перманентные характеристики мира).

### 6.2. Case Sensitive (Учёт регистра)

| Значение | Поведение |
|---|---|
| ON (включено) | Строгое соответствие регистру. "Цветочные" и "цветочные" — РАЗНЫЕ ключи. "NIKS" и "никс" — разные. |
| OFF (выключено, рекомендуется) | Регистр игнорируется. "ЦВЕТОЧНЫЕ", "цветочные", "Цветочные" — один и тот же ключ. |

**Рекомендация:** Оставлять **OFF** для 99% записей. Включать ON только если нужно различать собственные имена и обычные слова с совпадающим написанием (редкий кейс).

### 6.3. Whole Words (Целые слова)

| Значение | Поведение |
|---|---|
| ON (включено) | Ищется точное совпадение ЦЕЛОГО слова. Ключ "кот" не сработает на "скот", "котлета", "который". |
| OFF (выключено) | Ищется подстрока в любом месте. Ключ "кот" сработает на "скот", "котлета", "который". |

**Карта настроек по подходам:**

| Подход | Case Sensitive | Whole Words |
|---|---|---|
| Подход 1 (все формы) | OFF | ON |
| Подход 2 (суффиксный) | OFF | OFF |
| Подход 3 (regex) | OFF | ON (границы в regex) |

### 6.4. Group Scoring (Групповое скорирование)

**Определение:** Система начисления очков за совпадения внутри группы lorebook. Каждое совпадение ключа = 1 очко.

**Механика:**
- Записи внутри одной группы ранжируются по количеству очков (чем больше совпадений, тем выше приоритет).
- При ограниченном бюджете токенов context entries с наибольшим счётом вставляются первыми.
- 1 совпадение = 1 очко (независимо от того, какой именно ключ сработал).

**Когда включать:**
- Включайте Group Scoring, если в одной группе есть несколько записей, конкурирующих за место в контексте, и нужно приоритизировать наиболее релевантные.
- Выключайте, если все записи группы равнозначны или используете insertion_order для ручной приоритизации.

### 6.5. Сводная таблица настроек

| Параметр | Значение по умолчанию | Рекомендация | Описание |
|---|---|---|---|
| Scan Depth | 4 | 2-4 для сцен, 1 для реактивного | Глубина сканирования истории |
| Case Sensitive | false | OFF (false) | Учёт регистра символов |
| Match Whole Words | true | Зависит от подхода | Поиск целых слов vs подстрок |
| Use Group Scoring | false | OFF для простых случаев | Ранжирование по количеству совпадений |

## 7. Двуязычные ключи (EN + RU)

### 7.1. Принцип

Каждая запись lorebook, рассчитанная на билингвальный ролеплей (или на переключение между языками), должна содержать ключи **И для английского, И для русского языка** в одном массиве ключей. Это гарантирует срабатывание независимо от того, на каком языке пишет пользователь или бот.

### 7.2. Формат английского ключа

Для английского языка границы слова `\b` работают корректно:

```javascript
\bWord\b/i
```

| Компонент | Значение |
|---|---|
| `\b` | Граница слова (word boundary) — работает для латиницы |
| `Word` | Искомое слово (корень или полная форма) |
| `\b` | Закрывающая граница слова |
| `/i` | Флаг ignoreCase — игнорирование регистра |

**Примеры EN-ключей:**
```
\bflower\b/i
\bfield\b/i
\bforget-me-not\b/i
\bpoppy\b/i
\bsunset\b/i
\bray\b/i
\bcat\b/i
\bNix\b/i
```

### 7.3. Формат русского ключа

Для русского языка `\b` НЕ работает (кириллические символы не поддерживают word boundaries в JS). Используется lookaround:

```javascript
(?:^|[^а-яёА-ЯЁ])Word[а-яёА-ЯЁ]*(?=[^а-яёА-ЯЁ]|$)/i
```

**Примеры RU-ключей:**
```
(?:^|[^а-яёА-ЯЁ])цветочн[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i
(?:^|[^а-яёА-ЯЁ])незабуд[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i
(?:^|[^а-яёА-ЯЁ])мак[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i
```

### 7.4. Объединённый массив ключей

В JSON-формате lorebook ключи хранятся как массив строк. EN и RU паттерны объединяются в один массив:

```json
{
  "key": [
    "\\bflower\\b/i",
    "\\bfield\\b/i",
    "\\bforget-me-not\\b/i",
    "\\bpoppy\\b/i",
    "\\bsunset\\b/i",
    "\\bray\\b/i",
    "\\bcat\\b/i",
    "\\bNix\\b/i",
    "(?:^|[^а-яёА-ЯЁ])цветочн[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i",
    "(?:^|[^а-яёА-ЯЁ])пол[ьяеюйа-яё]*(?=[^а-яёА-ЯЁ]|$)/i",
    "(?:^|[^а-яёА-ЯЁ])незабуд[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i",
    "(?:^|[^а-яёА-ЯЁ])мак[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i",
    "(?:^|[^а-яёА-ЯЁ])закатн[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i",
    "(?:^|[^а-яёА-ЯЁ])луч[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i",
    "(?:^|[^а-яёА-ЯЁ])кот[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i",
    "(?:^|[^а-яёА-ЯЁ])никс[а-яё]*(?=[^а-яёА-ЯЁ]|$)/i"
  ]
}
```

### 7.5. Рекомендации по билингвальным ключам

- Всегда включайте **ОБА языка** даже если ролеплей ведётся на одном — бот может переключиться.
- Для EN используйте `\b...\b/i` — это стандарт и он работает надёжно.
- Для RU всегда используйте lookaround `(?:^|[^а-яёА-ЯЁ])...(?=[^а-яёА-ЯЁ]|$)` — `\b` с кириллицей даёт непредсказуемые результаты.
- Порядок ключей в массиве не важен для срабатывания — все проверяются параллельно.
- Для regex-ключей флаг `/i` дублирует функционал Case Sensitive = OFF — достаточно одного из них.

## 8. Рабочий процесс выбора подхода

### 8.1. Алгоритм решения

```
1. Запись содержит 1-3 простых слова без сложной морфологии?
   → Подход 1: простой список всех форм
   
2. Запись содержит много словоформ / сложную морфологию?
   └─→ Вы знакомы с regex?
       ├─→ ДА → Подход 3: regex (оптимальный)
       └─→ НЕТ → Подход 2: суффиксный + Secondary Filters
       
3. Запись критически важна и не должна иметь ложных срабатываний?
   → Подход 1 с полным списком форм
   
4. Запись описывает уникальную сущность (имя, топоним)?
   → Подход 3: regex с точными границами
   
5. Ролеплей билингвальный (RU + EN)?
   → Подход 3 + двуязычные ключи (раздел 7)
```

### 8.2. Рекомендации по контексту

| Тип записи | Рекомендуемый подход | Обоснование |
|---|---|---|
| Уникальное имя персонажа | Regex (Подход 3) | Минимум ключей, максимум точности |
| Общая локация | Regex + Secondary | Покрывает формы + исключает ложные контексты |
| Простой предмет (мало форм) | Подход 1 | Быстрее написать формы, чем regex |
| Временной статус / эффект | Подход 1 или 3 | Зависит от количества связанных слов |
| Глобальный миро-лор | Regex + Scan Depth=All | Точность при масштабе |

## 9. Таблица типичных ошибок

| № | Ошибка | Последствие | Решение |
|---|---|---|---|
| 1 | Ключ "в" (предлог) | Срабатывание в КАЖДОМ сообщении, полностью обесценивает запись | Удалить, заменить на конкретное существительное |
| 2 | Ключ "он" / "она" | То же — полная деградация полезности | Удалить, использовать имена или конкретные термины |
| 3 | Whole Words = ON + кириллический regex без lookaround | Непредсказуемые срабатывания, границы слов не работают | Использовать `(?:^|[^а-яёА-ЯЁ])...(?=[^а-яёА-ЯЁ]|$)` |
| 4 | Whole Words = OFF + короткий корень "пол" | Ложные срабатывания на "политика", "полный", "аполлон" | Перейти на regex или добавить Secondary NOT ANY с ложными словами |
| 5 | Фраза-ключ "цветочные поля" без regex | Не срабатывает при "поля цветочные" или "цветочные луга и поля" | Перейти на regex с `цветочн` + `пол` как отдельные паттерны |
| 6 | Case Sensitive = ON без необходимости | Пропуск срабатываний при смене регистра (начало предложения, капс) | Case Sensitive = OFF (по умолчанию) |
| 7 | Scan Depth = 1 для фонового лора | Лор исчезает из контекста после одного сообщения | Увеличить Scan Depth до 3-4 |
| 8 | Scan Depth = All для реактивного лора | Перегрузка контекста, срабатывание на старые сообщения | Уменьшить Scan Depth до 1-2 |
| 9 | Рекурсия без планирования ключей | Зацикливание, каскадная активация, переполнение контекста | Планировать ключи от конца цепочки к началу |
| 10 | Имя пользователя в ключах | Срабатывание при каждом обращении к пользователю | Использовать контекстные слова вместо имён |
| 11 | HTML-теги / {{char}} в ключах | Скрытые срабатывания на техническую разметку | Исключить системные маркеры из ключей |
| 12 | Secondary Filters без понимания логики | Непредсказуемое поведение — лор не срабатывает или блокируется ошибочно | Перечитать раздел 5, использовать таблицу логики |
| 13 | Отсутствие RU-ключей для русского ролеплея | Лор не срабатывает на русском тексте | Всегда добавлять русские паттерны для RU-ролеплея |
| 14 | `\b` в regex для русских слов | Некорректная работа границ, ложные срабатывания или пропуски | Заменить на `(?:^|[^а-яёА-ЯЁ])...(?=[^а-яёА-ЯЁ]|$)` |
| 15 | Слишком много ключей (>15) на одну запись | Перегрузка системы, шум, сложность отладки | Сократить до 10, использовать regex для покрытия форм |

## Приложение A: Быстрый справочник по regex-шаблонам

### Шаблоны для русского языка

```
// Базовый шаблон (все формы слова):
(?:^|[^а-яёА-ЯЁ])КОРЕНЬ[а-яё]*(?=[^а-яёА-ЯЁ]|$)

// Для слов с чередованием корня (поле/поля):
(?:^|[^а-яёА-ЯЁ])пол[ьяеюйа-яё]*(?=[^а-яёА-ЯЁ]|$)

// Для имён собственных (точное совпадение + падежи):
(?:^|[^а-яёА-ЯЁ])никс[а-яё]*(?=[^а-яёА-ЯЁ]|$)

// Для составных имён (кот никс — два паттерна):
(?:^|[^а-яёА-ЯЁ])кот[а-яё]*(?=[^а-яёА-ЯЁ]|$)
(?:^|[^а-яёА-ЯЁ])никс[а-яё]*(?=[^а-яёА-ЯЁ]|$)
```

### Шаблоны для английского языка

```
// Простое слово (все формы):
\bword\b/i

// Корень с wildcard (word, words, wording):
\bword\w*\b/i

// Несколько вариантов:
\b(colour|color)\b/i
```

### Разница EN vs RU шаблонов

| Аспект | EN | RU |
|---|---|---|
| Границы слова | `\b` работает | `\b` НЕ работает |
| Альтернатива границам | `\b` | `(?:^|[^а-яёА-ЯЁ])` + `(?=[^а-яёА-ЯЁ]|$)` |
| Символьный класс | `\w` (латиница + цифры + _) | `[а-яё]` (кириллица) |
| Флаг регистра | `/i` | `/i` или Case Sensitive = OFF |
